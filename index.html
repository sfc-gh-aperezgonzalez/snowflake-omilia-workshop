Anonymize with AI_REDACT

12-20 min

Remove PII from transcripts before they can be used for training or analytics. AI_REDACT automatically detects and masks names, emails, phone numbers, and addresses.

Production Note In production, this step could be replaced by a call to Omilia's own anonymization model/logic via an external function or UDF. AI_REDACT is used here as a built-in, zero-setup alternative. 
CREATE OR REPLACE TABLE CALLS_ANON AS
SELECT
    CALL_ID, CUSTOMER_ID, QUEUE, LANG,
    CALL_START_TS, CALL_END_TS, DURATION_SEC,
    AGENT_ID, CHANNEL,
    AI_REDACT(RAW_TRANSCRIPT) AS TRANSCRIPT_ANON
FROM CALLS_RAW;
-- Compare: raw vs anonymized
SELECT
    r.CALL_ID,
    LEFT(r.RAW_TRANSCRIPT, 100) AS original,
    LEFT(a.TRANSCRIPT_ANON, 100) AS anonymized
FROM CALLS_RAW r
JOIN CALLS_ANON a ON r.CALL_ID = a.CALL_ID
LIMIT 5;
PII replaced with placeholders like [NAME], [EMAIL], [PHONE_NUMBER].
3

Annotation Write-Back

20-30 min

Simulate what happens when a human annotation team labels call intents. In production, an external tool (Label Studio, Prodigy, or Omilia's internal UI) writes labels back into Snowflake via API. Here we use AI_CLASSIFY to produce semantically meaningful labels — exactly what a human annotator would assign based on transcript content.

Why AI_CLASSIFY? Using AI to simulate annotations produces content-based labels (not random ones). When we train a model on these in Step 5, the training is realistic — the model learns actual patterns between call content and intent. 
-- Simulate human annotations using AI_CLASSIFY
CREATE OR REPLACE TABLE HUMAN_ANNOTATIONS AS
SELECT
    CALL_ID,
    AI_CLASSIFY(
        TRANSCRIPT_ANON::VARCHAR,
        ['billing_dispute', 'service_outage', 'plan_upgrade', 'device_issue', 'account_closure', 'general_inquiry']
    ):labels[0]::VARCHAR AS INTENT_LABEL,
    'annotator_' || MOD(ABS(HASH(CALL_ID)), 5) AS ANNOTATOR_ID,
    DATEADD(hour, -MOD(ABS(HASH(CALL_ID)), 48), CURRENT_TIMESTAMP()::TIMESTAMP_NTZ) AS ANNOTATED_AT,
    ROUND(0.85 + (MOD(ABS(HASH(CALL_ID)), 15) / 100.0), 2) AS CONFIDENCE
FROM CALLS_ANON
WHERE MOD(ABS(HASH(CALL_ID)), 3) = 0;

SELECT INTENT_LABEL, COUNT(*) AS cnt
FROM HUMAN_ANNOTATIONS
GROUP BY INTENT_LABEL ORDER BY cnt DESC;
-- Merge: combine anonymized data with annotation labels
CREATE OR REPLACE TABLE CALLS_TRAINING_READY AS
SELECT
    a.*,
    h.INTENT_LABEL,
    h.ANNOTATOR_ID,
    h.ANNOTATED_AT,
    h.CONFIDENCE AS ANNOTATION_CONFIDENCE,
    CASE WHEN h.CALL_ID IS NOT NULL THEN 'human' ELSE 'model' END AS LABEL_SOURCE
FROM CALLS_ANON a
LEFT JOIN HUMAN_ANNOTATIONS h ON a.CALL_ID = h.CALL_ID;

-- Verify: count by label source
SELECT LABEL_SOURCE, COUNT(*) AS cnt
FROM CALLS_TRAINING_READY
GROUP BY LABEL_SOURCE;
~65 rows with LABEL_SOURCE = 'human' (AI_CLASSIFY-based intent labels), rest with 'model'.

Create Immutable ML Dataset

30-35 min

Freeze the training-ready data into an immutable, versioned ML Dataset — a "recipe" that guarantees reproducibility. If you retrain 6 months from now, you can prove exactly which data was used.

Why Datasets? Tables change. Datasets don't. Each version is a frozen snapshot with its own lineage. Your team can maintain multiple recipes: v1 = all annotated calls, v2 = Greek-only calls, v3 = billing queue only. 
CREATE DATASET IF NOT EXISTS CALLS_TRAINING_DATASET;

ALTER DATASET CALLS_TRAINING_DATASET ADD VERSION 'v1'
  FROM (SELECT * FROM CALLS_TRAINING_READY)
  COMMENT = 'All annotated calls - initial training recipe';

-- Verify: list all versions
SHOW VERSIONS IN DATASET CALLS_TRAINING_DATASET;
Dataset CALLS_TRAINING_DATASET with version v1 containing your annotated training data.

Train & Register a Model

35-45 min

Train an intent classifier on the annotated data using Snowpark ML, then register it in the Model Registry. This completes the lineage chain: Source Table → Dataset → Model.

Production Note This is a minimal example to show the pattern. In production you'd use the full MLOps stack: Feature Store, Experiments, Model Explainability (SHAP), and ML Observability for drift detection. 
# Switch to Python cell in your notebook
from snowflake.snowpark.context import get_active_session
from snowflake.ml.modeling.ensemble import RandomForestClassifier
from snowflake.ml.registry import Registry
from snowflake.ml.dataset import load_dataset

session = get_active_session()

# Load from the immutable dataset
ds = load_dataset(session, "CALLS_TRAINING_DATASET", "v1")
train_df = ds.read.to_snowpark_dataframe()

# Only use human-annotated rows
labeled_df = train_df.filter(train_df["LABEL_SOURCE"] == "human")
print(f"Training on {labeled_df.count()} human-annotated rows")

# Train intent classifier
model = RandomForestClassifier(
    n_estimators=50,
    input_cols=["DURATION_SEC"],
    label_cols=["INTENT_LABEL"],
)
model.fit(labeled_df)

# Register in Model Registry
reg = Registry(session=session)
mv = reg.log_model(
    model,
    model_name="INTENT_CLASSIFIER",
    version_name="v1",
    sample_input_data=labeled_df.select("DURATION_SEC").limit(10),
    comment="Intent classifier trained on human-annotated call data"
)
print(f"Model registered: {mv.model_name} version {mv.version_name}")
Model INTENT_CLASSIFIER version v1 registered in the Model Registry.

Batch Inference — Prove It Works

45-50 min

Run the registered model on all calls to generate predictions. This proves the end-to-end pipeline delivers real value — you can compare predicted vs actual labels and measure accuracy.

CREATE OR REPLACE TABLE CALLS_PREDICTIONS AS
SELECT
    CALL_ID, QUEUE, LANG, DURATION_SEC, CHANNEL, INTENT_LABEL, LABEL_SOURCE,
    INTENT_CLASSIFIER!PREDICT(DURATION_SEC):OUTPUT_INTENT_LABEL::VARCHAR AS PREDICTED_INTENT_LABEL
FROM CALLS_TRAINING_READY;

-- Compare predictions vs actual labels
SELECT
    CALL_ID, QUEUE, DURATION_SEC,
    INTENT_LABEL AS ACTUAL_LABEL,
    PREDICTED_INTENT_LABEL,
    CASE WHEN INTENT_LABEL = PREDICTED_INTENT_LABEL THEN 'correct' ELSE 'mismatch' END AS MATCH,
    LABEL_SOURCE
FROM CALLS_PREDICTIONS
WHERE LABEL_SOURCE = 'human'
LIMIT 15;
CALLS_PREDICTIONS table with predicted intent labels. Compare actual vs predicted to see model accuracy.

Semantic View & Cortex Agent

Bonus

Create a Semantic View for natural-language querying, then a Cortex Agent that answers questions about your data using text-to-SQL.

8A: Create Semantic View

CREATE OR REPLACE SEMANTIC VIEW CALLS_SEMANTIC_VIEW

  TABLES (
    calls AS CALLS_TRAINING_READY
      PRIMARY KEY (CALL_ID)
      COMMENT = 'Omilia call center training-ready data'
  )

  DIMENSIONS (
    calls.queue AS QUEUE
      COMMENT = 'Call queue: billing, technical_support, cancellations, sales, general',
    calls.lang AS LANG
      COMMENT = 'Language of the call: en, el, es',
    calls.channel AS CHANNEL
      COMMENT = 'Channel: phone, virtual_assistant, chat',
    calls.agent_id AS AGENT_ID
      COMMENT = 'Agent who handled the call',
    calls.call_start_ts AS CALL_START_TS
      COMMENT = 'Call start timestamp',
    calls.intent_label AS INTENT_LABEL
      COMMENT = 'Annotated intent: billing_dispute, service_outage, plan_upgrade, device_issue, account_closure, general_inquiry',
    calls.label_source AS LABEL_SOURCE
      COMMENT = 'Who provided the label: human or model'
  )

  METRICS (
    calls.total_calls AS COUNT(CALL_ID)
      COMMENT = 'Total number of calls',
    calls.avg_duration AS AVG(DURATION_SEC)
      COMMENT = 'Average call duration in seconds',
    calls.human_annotated_pct AS AVG(CASE WHEN LABEL_SOURCE = 'human' THEN 1.0 ELSE 0.0 END)
      COMMENT = 'Percentage of calls with human annotations (0-1)'
  )

  COMMENT = 'Semantic view over Omilia annotated call data for natural-language analytics';

8B: Create Cortex Agent

CREATE OR REPLACE AGENT CALLS_ANALYST_AGENT
  COMMENT = 'Omilia Call Center Analyst — ask questions about call volumes, intents, annotation coverage, and queue performance'
  FROM SPECIFICATION
  $$
  instructions:
    response: |
      You are the Omilia Call Center Analyst. Answer questions about call center operations clearly and concisely.
      When showing data with categories or comparisons, always include a chart visualization.
      Use bar charts for category comparisons and line charts for time trends.
      Format numbers with appropriate precision (percentages to 1 decimal, scores to 2 decimals).

  tools:
    - tool_spec:
        type: "cortex_analyst_text_to_sql"
        name: "CallsAnalyst"
        description: "Queries Omilia call center data including call volumes by queue, intent distribution, annotation coverage, duration patterns, and channel breakdowns."

  tool_resources:
    CallsAnalyst:
      semantic_view: "CALLS_SEMANTIC_VIEW"
      execution_environment:
        type: "warehouse"
        warehouse: "WORKSHOP_WH"
  $$;

Test Your Agent
